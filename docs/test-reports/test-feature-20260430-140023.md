## Per-Feature Details

### End-to-end problem solving (onboard → heisenberg → DMRG → verification → handoff)

- **Role:** First-year PhD student, two months in. One semester of grad solid-state, no many-body theory. Knows what a Hamiltonian is; has heard "spin liquid" and "DMRG" but couldn't define precisely. Impatient, picks option 1, doesn't read docs.
- **Use Case:** Compute `E/N` for 1D Heisenberg chain N=20 S=1/2 AFM OBC; baseline for advisor's frustrated-spin project.
- **What they tried:** Read README, onboard skill, heisenberg skill, dmrg method card, limits/symmetry/benchmark KB cards. Simulated the full conversation an agent would have given the harness's prescriptions.
- **Discoverability:** Yes. README is short and scannable. Onboard skill exists for "where do I start." Heisenberg skill matches the problem cleanly.
- **Setup:** Makefile installs work as documented (`make install julia && make install itensors`).
- **Functionality:** The agent recommends DMRG, runs it, verifies against limits + symmetry + a benchmark, handles a pushback turn correctly, proactively offers research-continuation next steps. Simulated transcript runs 8 turns to a verified result without the persona being overwhelmed.
- **Expected vs Actual Outcome:** Met. The pattern works: agent recommends, student ratifies, work happens. Persona never had to choose a method, defend a question, or read documentation.
- **Blocked steps:** None.
- **Friction points (4 specific):**
  1. **`heisenberg` Diagnose section is a trap for a literal-minded agent.** It lists 8 things to fix; the "infer defaults and state them" rescue clause is buried at the end. An agent following the bullet list literally would hit the persona with 8 questions and lose them.
  2. **`onboard` step 2's question is too long.** The comma-separated list of 8 model names + 5 physics topics reads like a menu / registration form, not a conversation.
  3. **Benchmark card precision manufactures a fake discrepancy.** `N=20 OBC E ≈ -8.6826` (rounded) vs DMRG result `-8.6824` looks like an error to the persona; ED gives `-8.68247` exactly.
  4. **Heisenberg skill has no proactive "Next steps" section.** The simulated agent's A7 (proposing TD-limit extrapolation as a research continuation) was implied by AGENTS.md, not prescribed by the skill itself. Without prescription, that beat is fragile.

- **Where the pattern works best (specific):** A1 (single decision surfaced: verification depth, with recommendation); A4 (genuine reconsideration on user pushback, two real options); A6 (three real next steps including a non-padded "stop here").
- **Doc suggestions:** see Suggestions section below.

## Expected vs Actual Outcome

The fake-steering-wheel pattern holds for an impatient junior persona on the canonical entry-level problem. The agent drives, the student ratifies, the work gets done. Decisions are surfaced only when they actually matter (verification depth at A1; convention check at A4; research direction at A6).

The pattern's stress points are the literal-minded-agent risk in `heisenberg`'s Diagnose section, the menu-shaped onboard question, and a benchmark precision detail.

## Issues Found

Severity ordered:

1. **medium** — `heisenberg` Diagnose section orders information such that a literal agent asks 8 questions before doing anything. The defaults-and-state-assumption rule should be the headline, not the footer.
2. **medium** — `onboard` step 2 lists 13 named topics in one question. Reads like a registration form to an impatient user.
3. **low** — `benchmark-numbers.md` 1D Heisenberg N=20 entry rounds to `-8.6826` when ED gives `-8.68247`. The 2e-4 discrepancy with a DMRG run looks like an error.
4. **low** — `heisenberg` skill omits a "Next steps" section. Research-continuation prompts (TD limit, gap, frustrated follow-on, stop) currently rely on AGENTS.md's Writeup handoff convention plus agent intuition. Brittle.
5. **low** — Cross-method validation in `heisenberg` is generic ("DMRG ↔ ED"); the persona's exact path (N=20 OBC) is the perfect canonical example and should be named in the skill so the agent always reaches for it on first contact.

## Suggestions

Actionable, ordered by impact:

1. **`tools/skills/problems/models/heisenberg/SKILL.md` Diagnose section** — invert order. Lead with: "If only 'Heisenberg' is given, assume S=1/2, isotropic, NN, AFM, OBC; state the assumption in one line." Move the 8-item list to "ask only if the user contradicts the defaults."

2. **`tools/skills/onboard/SKILL.md` step 2** — replace the comma-separated 13-name list with: "Tell me the model or question in your own words — I'll route to the right skill." Keep the full list available in a follow-up only if the user's first answer doesn't match anything.

3. **`knowledge-base/benchmark-numbers.md` 1D Heisenberg row** — change `N=20 OBC E ≈ -8.6826 (E/N ≈ -0.4341)` to `N=20 OBC E = -8.68247 (ED, exact)`. Removes the manufactured 2e-4 discrepancy.

4. **`tools/skills/problems/models/heisenberg/SKILL.md`** — add a "Next steps" section after Verification, listing canonical research continuations: TD-limit extrapolation (`E/N` vs `1/N²`), spin gap, frustrated follow-on, stop. Mirrors the Writeup handoff but for research direction.

5. **`tools/skills/problems/models/heisenberg/SKILL.md` Verification** — name the canonical N=20 DMRG ↔ ED cross-method check explicitly so agents anchor to the right first-contact move.
